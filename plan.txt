================================================================================
FRAUDFLOW — DYNAMIC BACKEND REFACTOR PLAN
================================================================================
Date       : 2026-03-14
Branch     : feat/dynamic-backend  (branch off feat/connect-frontend)
Objective  : Remove every hardcoded magic number, duplicate constant, and
             mandatory override from the fraud-detection pipeline so that all
             risk weights, thresholds, and verdicts are driven by real runtime
             logic and a single, tunable configuration file.
================================================================================


────────────────────────────────────────────────────────────────────────────────
WHAT IS WRONG RIGHT NOW  (audit summary)
────────────────────────────────────────────────────────────────────────────────

1. The constant NEW_APP_AGE_HOURS (72 hours) is copy-pasted into 4 separate
   files: profiler.py, scope_rules.py, decision_engine.py, memory_store.py.
   They can drift out of sync silently.

2. All 7 composite risk-score weights (e.g. off_hours: 4.0, benford: 2.0) are
   buried in the middle of profiler.py with no label and no central reference.

3. The BLOCK/FLAG/ALLOW score thresholds (>= 6.0 / 3.5–5.9 / < 3.5) are
   hardcoded *inside a Python string* (the Claude system prompt). The prompt
   can fall out of sync with the real scoring logic with no error.

4. Two completely separate scoring systems exist:
   - profiler.py  — used by the main demo pipeline
   - scope_rules.py — used only by GET /api/profile/{app_id}
   They have different weight tables and return different scores for the same
   app. The frontend gets contradictory numbers depending on which route it
   calls.

5. decision_engine.py contains a _mandatory_verdict() function that bypasses
   Claude entirely and force-BLOCKs certain apps using hardcoded Python
   conditions. Claude is never actually deciding those verdicts.

6. TaxEasy's registration date is hardcoded to 2026-03-12. Once today's date
   passes ~March 15 2026, the new_app_risk signal stops firing and Scenario 3
   silently breaks.

7. The memory store resets to empty on every server restart. The first run of
   any scenario always returns "No similar patterns found." — meaningless
   context for the AI.

8. Trust-score penalties, API result limits, pipeline call window, and other
   operational numbers are scattered as unnamed literals across pipeline.py,
   read_routes.py, router.py, and telemetry.py.

================================================================================
Status  : [ ] pending
Branch  : feat/dynamic-backend
Commit  : "fix(seed): make TaxEasy registration date relative to prevent drift"

Description
-----------
TaxEasy's registered_at is hardcoded to datetime(2026, 3, 12, 8, 0, 0).

Once today's date passes roughly March 15 2026 (72 hours after that date),
_app_age_hours exceeds NEW_APP_AGE_HOURS and the new_app_risk signal stops
firing. Scenario 3's BLOCK verdict depends on this signal. The scenario would
silently produce weaker results with no error or warning.

Changes
-------
  File: backend/app/seed.py

  1. Add `from datetime import datetime, timedelta` at the top if not present.

  2. Replace line 58:
       registered_at=datetime(2026, 3, 12, 8, 0, 0),
     With:
       registered_at=datetime.utcnow() - timedelta(hours=48),

     This guarantees TaxEasy is always exactly 48 hours old when seeded,
     regardless of when seed.py is run. new_app_risk always fires.

  No other changes to seed.py in this task.

Acceptance criteria
-------------------
  - TaxEasy's registered_at is computed at runtime, not a fixed literal
  - Re-running seed.py on any date still produces a TaxEasy that is 48h old
  - BudgetBuddy and QuickPay registration dates are unchanged (they are
    intentionally historical and do not affect any risk signals)
